## A causa-raiz (L-C)
`extractText` devolve **um único `text` concatenado**; o parser **re-regexa esse blob**. Quando a análise roda, **a fonte de cada dado já se perdeu** → não dá para dizer "isto veio de JSON-LD do produto" vs "isto é prosa". Por isso (1) proveniência na UI (P5) segue impossível, (2) o risco-vizinho é combatido por heurística de prosa em vez de **escopo na origem**, e (3) consertamos sintoma a sintoma. P2.4 conserta a **forma**, não mais um sintoma.

## A mudança (incremental, NÃO big-bang)
O extrator passa a emitir, além do `text` (fallback), **candidatos estruturados por campo**:

```ts
type FieldSource = "structured" | "meta" | "visible-text" | "reader";
type FieldScope  = "product" | "page" | "catalog";
interface Candidate<T> { value: T; source: FieldSource; scope: FieldScope; raw: string; }
// ExtractResult ganha (opcional, aditivo):
candidates?: {
  fiber?: Candidate<string>[];
  gsm?: Candidate<number>[];
  weave?: Candidate<Weave>[];
  // imagem já tem identidade/escopo (G1); pode entrar aqui depois
};
```
- **Fontes** já existem no código (JSON-LD `Product` = `structured`+`product`; `<meta>` = `meta`+`page`; containers visíveis = `visible-text`+`page`; reader markdown = `reader`+`page`). P2.4 **rotula** o que já coletamos, em vez de jogar no blob.
- **`scope` vem do A3/L-B:** candidato de nó escopado ao produto (JSON-LD Product, JSON island casado por handle/SKU) = `product`; resto = `page`/`catalog`. **Escopo fecha o risco-vizinho na origem** (não por rejeição-de-prosa).
- **Parser consome candidatos** por **precedência**: `structured+product` → `meta` → `visible-text` → `reader`; cai no **blob (fallback)** quando não há candidato. Carrega a fonte no contrato: `findings.<campo>.source`.

## Migração campo a campo (de-risca)
Um campo por vez, **mantendo o blob como fallback** e o **corpus congelado** como rede:
1. **fiber/composição** primeiro (maior valor, mais fontes estruturadas).
2. **gsm**.
3. **weave**.
4. (imagem já normalizada no G1 — integrar por último, se valer.)

A cada campo: adicionar a emissão de candidatos na fonte → fazer o parser preferir o candidato → **re-baseline do corpus auditado à mão** (diff campo a campo) → só então o próximo. Se um campo regredir o corpus, para e revê.

### 1b.1 — Decisão de design: NOVO extrator escopado ao lado, **não reescrever** `collectMaterialValues`
A adenda falou em "reescrever `collectMaterialValues`". **Refino para menor risco:** **manter `collectMaterialValues`/`embeddedJsonText` intactos** (seguem alimentando o blob `text` exatamente como hoje → **zero regressão**, Gap/Armani preservados) e **adicionar** um extrator escopado separado (`embeddedFiberCandidate`) que produz **só** o candidato `scope:"product"`. Aditivo e reversível (remover a função volta ao estado de 1a). O "rastrear o nó de origem" que a adenda pediu é feito na função nova, sem tocar o caminho do blob.

### 1b.2 — Algoritmo de escopo (propagação de identidade com **reset** em filho não-casado)
Tokens do produto exibido: reusar **`productUrlTokens(url)`** (G1) — handle/último segmento (≥4 chars) + grupos de dígitos (≥4) = SKU/productId.
Walk no JSON de cada `<script application/json>`/`__NEXT_DATA__`:
```
walk(node, tokens, inMatched, out):
  array → walk cada item com o mesmo inMatched
  object:
    id = identityOf(node, tokens)         // "match" | "nomatch" | "none"
    matched = id==="match" ? true
            : id==="nomatch" ? false      // ← filho com OUTRA identidade reseta (mata related-products aninhados)
            : inMatched                    // sem identidade → herda do ancestral
    if matched: coletar a material PRÓPRIA do node (mesma lógica de label/spec-row do collectMaterialValues)
    para cada valor: walk(valor, tokens, matched, out)
```
- `identityOf` (corrigido pelo Risco 1, confirmado ao vivo): **só `handle`/`slug` exato confere `product`.** Dados reais da Norse: `product_id`/`variant.id` são números de 14 dígitos **repetidos em cada variante**, e uma URL Shopify padrão `?variant=56535173103947` injeta esse número como token → um nó vizinho que **referencie** o mesmo variant-id casaria por engano e venceria o blob. Portanto: **match por grupo-de-dígitos/`id`/`sku` sozinho NÃO basta** → trata-se como `none` (não promove), a menos que o `handle`/`slug` também corrobore. `handle`/`slug` (não-numéricos, únicos por produto) são o sinal forte; dígitos são ruidosos. Chaves consideradas para o handle/slug: `handle`, `slug`. **Nunca** `name`/`title` (prosa).
- **Reset em `nomatch`** é o que fecha o vizinho aninhado **dentro** do produto (ex.: `{handle:matched, relatedProducts:[{handle:B, material:poly}]}` → B reseta → poly não coletado).
- Sem tokens (URL sem handle/dígitos) → **não** emite candidato (conservador; blob como hoje).

### 1b.3 — Integração (em `extractText`)
Adicionar, **depois** do candidato `jsonld`: `embeddedFiberCandidate($, url)` → se retornar material casada, `{ raw, source:"structured", scope:"product" }`. Ordem final dos candidatos fiber: **`jsonld`(struct/prod) → `embedded`(struct/prod) → `meta` → `visible`**. JSON-LD vence empate (mais padronizado). O parser já normaliza o `raw` (correção do 1a) e descarta `catalog`.

### 1b.4 — Benefício além de proveniência: **reduz** o risco-vizinho
Quando o candidato escopado casa, ele **vence o blob** no `parse()`. Hoje o blob contém material de vizinhos do JSON island (risco pré-1b). Com 1b, para o produto casado o veredito passa a vir do **material escopado**, não do blob possivelmente contaminado. 1b é **melhoria** de honestidade, não só rótulo. (Para não-casados, igual a hoje — sem regressão.)
